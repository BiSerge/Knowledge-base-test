Следующий шаг: Определить к какой таблице на SQL Server’е этот запрос. Использование обработки НавигаторПоМетаданным показало, что РегистрСведений.Инт_НастройкаПравТекущая есть таблица _InfoRg8462. Сначала решили посмотреть индексы у этой таблицы. Уровень фрагментации индексов зашкаливал за все рекомендуемые рамки. Однако, Rebuild индексов ничего не дал.. То есть рекомендаций 1С оказалось недостаточно.

В мониторе активности SQL видим, что при попытках открытия проформ в топе Recent Expensive Queries висит запрос к таблице _InfoRg8462. Там же есть возможность посмотреть план запроса SQL, что и пытались сделать с самого начала, чтобы увидеть как 1С и SQL строят запросы. В плане запроса было видно, что запрос делает сканирование индекса по 2 полям вместо перехода, и занимает это 100% времени, и тут же в плане запроса сообщение: «Не хватает индекса».

Делаем индекс по подсказке: 

 /*
 Missing Index Details from ExecutionPlan1.sqlplan
 The Query Processor estimates that implementing the following index could improve the query cost by 100%.
 */


 USE [int_cf5_IN]
 GO
 CREATE NONCLUSTERED INDEX [_InfoRg8462_Add_By_Avl] 
 ON [dbo].[_InfoRg8462] ([_Fld8463RRef],[_Fld8465])
 INCLUDE ([_Fld8464_TYPE],[_Fld8464_RTRef],[_Fld8464_RRRef])
 GO

В результате - снижение задержки в 3,5 раза.

В таблице 3 поля таких же как и [_Fld8465], в которых хранятся сами права. По образу и подобию создаем индекс для [_Fld8466], и задержка снижается в 25 раз относительно начальных значений. Создание третьего индекса не дает дополнительных преимуществ, индексов уже достаточно.

Вывод: Методика повышения производительности системы безопасности сводится к вычислению таблицы соответствующей РегистрСведений.Инт_НастройкаПравТекущая и построению в ней индексов необходимых эффективного выполнения запроса.
